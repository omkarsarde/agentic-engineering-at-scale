Agentic Engineering at Scale

Modern agents are not a prompt wrapped in a loop. They are probabilistic decision-makers embedded inside ordinary software systems: APIs, stores, queues, policies, sandboxes, evaluators, deployment pipelines, and human organizations. Engineering them well requires understanding both the model and everything that constrains what the model can do.

This book develops that understanding from the reader’s existing knowledge of neural networks. It derives the language-model substrate, follows one request through inference and serving, builds retrieval and agent mechanisms from scratch, and then subjects those mechanisms to evaluation, security, reliability, and operational pressure.

The governing habit is simple:

Add autonomy only when measured value pays for its reliability, cost, security, and operator burden.

Chapter-title tags are durability signals: [F] marks a durable foundation, [S] marks a fast-moving state-of-the-art area that needs periodic re-verification, and [F+S] combines both.

Two routes, one spine

Route A — full foundations. Read Chapters 1–32 in order. Use Appendix A just in time when a chapter points to a mathematical, planning, control, or distributed-systems bridge.

Route B — applied agent engineering. Read Chapter 1, then Chapters 12–32 in order. The skipped model material is never assumed silently: the relevant chapters open with bounded backfill boxes or a self-contained primer.

What makes the book executable

Each chapter contains one integrated build. A reference artifact grows through small deltas, and each delta establishes a named invariant. We do not print three complete versions of the same agent loop. We build one loop, break it deliberately, and add only the code needed to close the observed gap.

Numeric claims are generated by executable Python. Architecture, sequence, state, lifecycle, and trust-boundary diagrams are stored as Mermaid. Dense dependency and lineage graphs use Graphviz. Bespoke spatial explanations use editable vector source and SVG. Every visual answers one question and is followed by the conclusion the reader should retain.

What stays current

The main chapters teach mechanisms that should remain useful even as products change. Model names, API versions, hardware SKUs, prices, benchmark status, protocol revisions, and legal effective dates live in explicitly dated Landscape 2026 boxes and Appendix C. Deleting those boxes must not break the conceptual spine.
