16.3 The missing boundary

A customer writes, “Order A-17 arrived damaged. Please make this right.” A language model can compose a sympathetic reply. Give the same model a lookup tool and it can retrieve the order. Give it a refund tool and it can propose a remedy. None of those additions answers the most important engineering question: what stands between the proposal and the movement of money?

Suppose the system prompt says, “Never refund more than 50 dollars without approval,” but the Python handler accepts any integer. The model emits amount_cents=999999, the caller invokes the handler, and the payment provider accepts it. The prompt expressed an intention; the executable path granted authority. The real limit was not 50 dollars. It was whatever the handler and payment account permitted.

That gap is where agent engineering begins. A model call returns a candidate output. An agent system interprets candidate outputs as proposals inside a code-owned control system. The proposal can be useful, malformed, repetitive, unaffordable, or unsafe. The surrounding system must remain correct under all five possibilities.

The words model, workflow, agent, harness, runtime, and platform are often used as if they named the same object. They do not:

  • A model maps context to a distribution over possible outputs. It has no direct authority over an external system.
  • A workflow has a control path chosen primarily by code. Models may fill individual steps, but code determines which step follows.
  • An agent lets a model choose at least some next actions or control transitions from a bounded action space.
  • A harness assembles context, exposes tools, manages a workspace, records events, and applies local policy around the loop.
  • A runtime schedules and persists runs, interrupts them, resumes them, streams events, and coordinates concurrent work.
  • A platform supplies shared identity, gateways, deployment, evaluation, observability, cost controls, and governance.

The figure is nested because ownership is nested. The loop is not the harness; the harness is not the runtime; the runtime is not the platform. A library may package several layers, but the engineering responsibilities remain distinct. When a framework advertises “memory,” for example, ask whether it means a serialized run checkpoint, a long-term user fact, or durable recovery after an external effect. Those are different contracts even when one product exposes all three.

Two rules travel through the rest of the book. First, prompt text is not enforcement. A restriction that must hold belongs in a schema, policy check, capability boundary, transaction, or human process. Second, autonomy is assigned per action, not per agent. One support agent may autonomously read order status, propose a refund, require approval to issue it, and be forbidden from changing a bank account. Calling the whole agent “level three” hides the authority that matters.

16.5 An agent as an executable specification

An informal definition such as “an LLM that uses tools in a loop” identifies the shape but not the contract. We need a definition that tells us what to test.

Let h_t be the history visible at step t: the original task, prior proposals, and correlated observations. The model or deterministic strategy samples a proposal

a_t \sim \pi_\theta(\cdot \mid h_t),

where \pi_\theta is the proposal policy parameterized by model weights \theta. A code-owned gate evaluates the proposal against current state s_t and caller context c:

G(s_t, a_t, c) \in \{\text{allow},\ \text{deny},\ \text{escalate}\}.

Only an allowed proposal reaches an executor. The executor returns an observation o_{t+1}, and a transition function computes the next explicit state:

s_{t+1} = \tau(s_t, a_t, o_{t+1}).

This gives a useful agent specification

\mathcal{A} = (S, A, O, \pi_\theta, G, E, \tau, B, \Phi),

where S is the state space, A the proposal space, O the observation space, E the effect executor, B the resource budget, and \Phi the set of success, failure, escalation, and termination predicates. A typed implementation does not prove every property of this tuple, but it makes the properties nameable.

The first version of the artifact defines proposals, observations, tools, explicit run state, and typed endings. Notice that ToolCall is named a proposal in its documentation. Describing an action does not confer permission to execute it.

# agent_loop.py — public contracts introduced once
@dataclass(frozen=True)
class ToolCall:
    """A model proposal; it has no authority by itself."""
    call_id: str
    name: str
    arguments: dict[str, Any]

@dataclass(frozen=True)
class Observation:
    """A correlated, typed result returned to the strategy."""
    call_id: str
    ok: bool
    kind: str
    content: Any

@dataclass
class RunState:
    """Serializable state visible at each decision boundary."""
    task: str
    observations: list[Observation] = field(default_factory=list)
    trace: list[dict[str, Any]] = field(default_factory=list)
    steps: int = 0
    cost_units: int = 0

A useful specification states invariants before implementation details. This chapter uses five:

  1. Effect authorization: every handler invocation corresponds to a proposal that passed schema validation and policy at the execution boundary.
  2. Correlation: every tool observation carries the call identifier of the proposal that caused it.
  3. Boundedness: the kernel stops after finite step and modeled-cost budgets, even when the proposal policy never chooses to stop.
  4. Typed endings: answering, exhausting a budget, and making no progress are different results that a caller can branch on.
  5. Outcome separation: “the model returned an answer” is not the same predicate as “the task succeeded.” An external verifier decides the latter.

The fifth invariant prevents a common measurement error. A polite final message can accompany an unchanged database. The loop should report that it reached an answer boundary; the evaluator should inspect the environment and decide whether the requested outcome occurred.

16.6 The proposal–gate–execute–observe kernel

The core loop has four verbs. Keeping them separate is more important than the choice of framework.

Propose. A strategy receives explicit state and returns either a ToolCall or a FinalAnswer. In a live adapter, the strategy serializes state into messages and parses the model’s tool call. In the deterministic fixture, it returns the same types directly.

Gate. Code verifies that the tool exists, arguments match the declared contract, and current policy permits this exact proposal. Validation answers “is this well formed?” Authorization answers “may this caller perform this effect now?” Neither can be delegated to model obedience.

Execute. Only the allowed path invokes a handler. Expected failures become typed observations. Unexpected programming faults should still fail loudly rather than being converted into plausible text.

Observe. The result, denial, or expected error is correlated to the proposal and appended to state. The strategy can adapt on its next decision.

sequenceDiagram
    autonumber
    participant S as Strategy / model
    participant K as Control kernel
    participant G as Validation + policy gate
    participant T as Tool handler
    participant X as External system
    S->>K: ToolCall(call_id, name, arguments)
    K->>G: Validate exact proposal in current context
    alt denied or malformed
        G-->>K: Denial(reason)
        K-->>S: Observation(call_id, denied, reason)
    else allowed
        G-->>K: Allow
        K->>T: Invoke typed handler
        T->>X: Read or effect
        X-->>T: Result / expected error
        T-->>K: Typed outcome
        K-->>S: Observation(call_id, kind, content)
    end

The denied branch is a normal control path. The handler does not receive the call and the model receives useful information. This is stronger than asking the model to reconsider before execution because the same component that grants authority evaluates the final arguments.

The implementation follows the diagram. This excerpt is the stable center of the artifact; later additions bound it without introducing another loop.

# agent_loop.py — run_agent(), proposal through observation
decision = strategy.decide(state)
state.steps += 1
state.cost_units += 2
state.trace.append({"event": "proposal", "value": repr(decision)})

if isinstance(decision, FinalAnswer):
    return RunResult(Stop.ANSWERED, decision.text, state)

tool = tools.get(decision.name)
error = "unknown tool" if tool is None else _validate(decision, tool)
allowed, reason = gate(decision, state) if error is None else (False, error)

if not allowed:
    observation = Observation(decision.call_id, False, "denied", reason)
    state.observations.append(observation)
    state.trace.append({"event": "gate", "allowed": False, "reason": reason})
    # ... bounded no-progress handling is added in Section 16.5 ...
else:
    content = tool.handler(**decision.arguments)
    observation = Observation(decision.call_id, True, "result", content)
    state.observations.append(observation)

The example uses an exact top-level name -> type schema so the mechanism is visible. Production tools normally use JSON Schema or an equivalent typed contract, including ranges, enumerations, nested objects, defaults, and explicit rejection of unknown fields. Schema validation still does not establish business authorization. amount_cents can be a valid integer and an impermissible refund.

Call identifiers matter as soon as calls can be parallel, retried, or streamed. Position in a list is not a stable identity under concurrency. A result without correlation can be attached to the wrong proposal and contaminate later decisions.

This in-process kernel establishes a narrow guarantee: within one Python process, a denied call cannot reach a registered handler through run_agent. It does not authenticate a tenant, protect a secret, bind a human approval, isolate untrusted code, survive a crash, or reconcile an ambiguous remote effect. Later chapters add those responsibilities at their canonical boundaries.

16.7 A denial trace: the action that must not happen

The fixture models a damaged order paid at 4,999 cents. Its refund_order handler appends a receipt to an effect log. The policy gate permits automatic refunds up to 5,000 cents. An adaptive strategy initially proposes 9,999 cents.

The interesting event is not a successful lookup. It is the missing receipt after the over-limit proposal:

Step Proposal or observation Gate Effect log
1 lookup_order(A-17) allowed empty
2 refund_order(A-17, 9999) denied: above authority empty
3 refund_order(A-17, 4999) allowed one receipt
4 final answer references the receipt not applicable still one receipt

The denial becomes an observation, not an exception that destroys the run. That lets the strategy reduce the proposal to the actual purchase amount. More important, adaptation happens without weakening the gate. The model is free to propose another value; code remains free to deny it.

The policy itself is intentionally small:

# fixture.py — gate(), value boundary for the teaching fixture
def gate(call: ToolCall, _: RunState) -> tuple[bool, str]:
    if call.name == "refund_order" and call.arguments["amount_cents"] > 5_000:
        return False, "refund exceeds automatic authority"
    return True, "policy permits proposal"

This proves one value-limit property and nothing broader. It does not verify that the caller owns A-17, that the currency is correct, that a previous refund has not occurred, that the policy version is current, or that 4,999 cents is a desirable remedy. A trustworthy example states those gaps instead of letting a short function borrow the reputation of a production authorization service.

A denial must also be designed as data. It needs a stable category, a safe explanation, and enough detail for a strategy or human to choose a legal next step. Returning a raw stack trace may leak internals. Returning only “failed” encourages blind retries. The Observation.kind field distinguishes a denial from a transient dependency failure and a permanent domain error.

The focused test makes the safety claim executable. It lowers the repeat threshold to one, submits an over-limit proposal, and checks the effect log:

# test_ch16_agent_loop.py — denied effect invariant
tools, gate, effects = fixture()
result = run_agent(
    "Refund A-17",
    SupportStrategy("chain", refund_cents=9_999),
    tools,
    gate,
    Limits(max_steps=5, repeated_denials=1),
)
assert result.stop == Stop.NO_PROGRESS
assert effects == []

The empty list is the decisive assertion. Testing that the model said it was denied would not test containment. The code asserts the state owned by the effect boundary.

16.8 Termination, budgets, and typed failure

A gate can contain an action while the loop still fails operationally. A strategy may repeat the same denied proposal forever, wander across harmless reads, or retry a timed-out dependency until the caller’s budget is gone. “The model will eventually stop” is not a termination condition.

Finite step and cost limits give the in-process kernel a simple liveness property: each iteration consumes at least one step, so the loop must return after at most max_steps decisions. The artifact also charges two synthetic cost units per strategy decision. Real systems use observed input/output tokens, tool charges, wall-clock deadlines, and sometimes energy or scarce-resource quotas. The counter’s unit is less important than its code-owned enforcement.

Repeated denials reveal one form of no progress. The artifact hashes the tool name and canonicalized arguments, counts repeated denied proposals, and ends with NO_PROGRESS when the threshold is reached. This is intentionally narrower than a semantic stuck detector. A strategy that alternates between two denied values can evade it; Exercise 3 asks you to close that gap without treating a legitimate transient retry as a loop.

stateDiagram-v2
    [*] --> Ready
    Ready --> Proposed: strategy decision
    Proposed --> Answered: final answer
    Proposed --> Denied: validation or policy rejects
    Proposed --> Executing: gate allows
    Denied --> Ready: new information and budget remain
    Denied --> NoProgress: repeat threshold reached
    Executing --> Observed: result / transient / permanent
    Observed --> Ready: budget remains
    Ready --> StepLimit: no step remains
    Ready --> CostLimit: next call exceeds budget
    Answered --> [*]
    NoProgress --> [*]
    StepLimit --> [*]
    CostLimit --> [*]

Typed endings let callers choose different recovery policies. ANSWERED can proceed to outcome verification. NO_PROGRESS can request a human decision with the last denial. STEP_LIMIT can return partial findings. COST_LIMIT can stop before admitting another paid call. Merging all four into a string such as “Agent failed” discards the state needed for safe recovery.

Expected tool failures are observations too. A timeout is usually transient: retry may be legal if the operation is idempotent and the deadline allows it. A missing order is permanent for the current identifier: blind retry wastes budget. A policy denial is neither; changing the proposal or escalating may help. The artifact catches only the expected exceptions used by the fixture:

# agent_loop.py — execution delta for typed expected failures
try:
    content = tool.handler(**decision.arguments)
    observation = Observation(decision.call_id, True, "result", content)
except TimeoutError as exc:
    observation = Observation(decision.call_id, False, "transient", str(exc))
except (KeyError, ValueError) as exc:
    observation = Observation(decision.call_id, False, "permanent", str(exc))

state.observations.append(observation)
state.trace.append({"event": "observation", "value": asdict(observation)})

Do not turn every exception into an observation. An assertion failure, corrupted registry, or programming error should normally stop the process and alert an operator. Otherwise the model may improvise around broken safety code.

The deadline in a production runtime also needs precise semantics. Checking elapsed time between calls prevents admission of another call after a deadline; it does not preempt a model or tool already in flight. Stronger cancellation requires timeouts propagated to the client, cooperative cancellation inside tools, and a plan for effects that may have completed remotely. Chapter 26 owns that durable-effect problem.

16.9 Workflows are strategies, not new loops

The same task can be organized by different control policies. Implementing a prompt chain, router, evaluator loop, and adaptive agent as four separate runtimes repeats message handling, tool dispatch, trace capture, and error handling. The duplicated machinery hides the policy difference and lets safety behavior drift across paths.

Keep the execution kernel fixed and vary only the strategy that selects the next proposal. Then every comparison shares the same gate, effect handler, budget accounting, and trace contract; the strategy is the only meaningful variable.

  chain [label="Prompt chain\nfixed next step"]
  router [label="Router\ncode/model selects branch"]
  eval [label="Evaluator–optimizer\nrevise until criterion"]
  adaptive [label="Adaptive agent\nmodel selects next action"]
  kernel [label="Shared kernel\npropose → gate → execute → observe", fillcolor="#e7f2e8", color="#3a7d44"]
  trace [label="Shared typed trace + budgets", fillcolor="#f4eee2", color="#9a6b1f"]
  chain -> kernel
  router -> kernel
  eval -> kernel
  adaptive -> kernel
  kernel -> trace
}

A prompt chain chooses a fixed sequence: extract fields, retrieve policy, draft answer. It is easy to test and should be the default when the path is known. A router selects one of several bounded branches, such as billing versus technical support. A parallel fan-out sends independent subproblems to workers and joins typed results; it is useful only when the subproblems are genuinely independent and write conflicts are controlled. An evaluator–optimizer alternates generation with an external criterion until the criterion passes or a budget ends. An adaptive agent lets the proposal policy choose among available tools based on observations.

The differences belong in a small strategy surface:

# agent_loop.py — the only behavior a strategy must provide
class Strategy(Protocol):
    """Choose the next proposal from explicit run state."""
    def decide(self, state: RunState) -> Decision: ...

The chapter fixture implements chain, router, evaluator, and adaptive modes as thin deterministic strategies. They all use the same tool registry, validation, gate, budgets, trace, and stop types. This makes a cost/reliability comparison meaningful: the measured delta comes from control policy rather than from four unrelated implementations.

Pattern Who chooses the next branch? Best fit Characteristic failure
Chain Code Stable, known sequence Cannot adapt to an unmodeled branch
Router Code or one bounded model decision Distinct task families Misrouting contaminates the entire branch
Parallel fan-out Code-owned scheduler Independent reads or analyses Cost growth and conflicting writes
Evaluator–optimizer Code owns loop; verifier decides acceptance Output with an executable rubric Weak verifier rewards cosmetic revision
Adaptive agent Model within a bounded action space Path depends on new observations Wandering, compounding error, larger attack surface

The autonomy slider is continuous. A system can use deterministic routing and an agentic substep, or an adaptive read phase followed by a deterministic write transaction. The engineering question is not “workflow or agent?” as a global label. It is “which transition benefits from model choice, and what evidence pays for that choice?”

16.10 ReAct, planning, and replanning

ReAct popularized an effective pattern: interleave reasoning with environment actions and observations rather than asking the model to produce a complete answer from its initial context. The engineering value is not that private reasoning text becomes trustworthy. The value is that each consequential step can be grounded in a new observation before the next proposal.

A production loop should therefore make actions and observations explicit. It need not expose or persist hidden chain-of-thought. A concise decision summary, cited evidence, and structured action are usually better operational records: they are smaller, easier to redact, and tied to observable behavior. Chapter 25 returns to the limits of chain-of-thought monitoring.

Planning adds a proposed future path. A plan is not an execution grant. It is a hypothesis about dependencies and useful next actions. Treat its steps as typed intentions, then apply the normal gate at execution time. This matters because state can change between planning and acting.

Three planning styles cover many systems:

Plan then execute. The model produces a dependency graph or ordered list once. Code executes eligible steps and verifies results. This works when the environment is stable and the cost of planning is small relative to execution.

Rolling plan. The model maintains a short horizon and revises after each observation. This reduces stale long plans but spends more model calls and can oscillate.

Planner–executor–verifier. Separate roles propose steps, execute through the same gate, and check outcomes. Separation can reduce correlated errors only when the verifier receives independent evidence or a different capability. Three prompts to the same model over the same context do not automatically create independence.

Replanning should be triggered by code-visible events: a precondition failed, a dependency changed, a verifier rejected an outcome, a budget crossed a threshold, or a human altered the goal. “Reflect whenever uncertain” gives neither the runtime nor evaluator a stable event to inspect.

Search over possible reasoning paths and verifier-guided allocation were developed in Chapter 8. Here the systems conclusion is narrower: any search procedure must consume an explicit budget, and a candidate plan must still pass the action gate one step at a time. More inference-time thought does not enlarge authority.

16.11 Execution-grounded verification

The kernel returns ANSWERED when the strategy emits a final answer. That is a control-flow fact, not a task-quality judgment. The build evaluates a refund task by inspecting the effect log for exactly one receipt with the intended order and amount. A response saying “Your refund is complete” receives a zero if no receipt exists.

Verifiers come in three useful layers:

  1. Invariant checks ask whether the system stayed inside its rules: no denied handler invocation, no unmatched call identifier, no budget overrun.
  2. Process checks ask whether required evidence or steps occurred: the order was read before a refund, the final answer cites the resulting receipt, the test command ran after a code edit.
  3. Outcome checks inspect the environment: database state, passing tests, a compiled artifact, account balance, or simulator goal state.

Outcome checks are strongest when the environment exposes a deterministic final-state predicate. Process checks are necessary when some outcomes are delayed or partially observable. Model-based judges can cover semantic qualities that code cannot express, but they need calibration and isolation from hostile candidate content. Chapter 22 treats that measurement problem in depth.

The chapter’s four tests form a small executable specification:

  • an over-limit refund creates no effect;
  • an adaptive strategy can use a denial as information and complete within policy;
  • a strategy that keeps reading stops at the step budget;
  • a final answer ends with ANSWERED and causes no effect by itself.

These tests point toward framework-independent operating principles:

  • Keep control flow and budgets in code.
  • Treat model outputs as untrusted typed proposals.
  • Keep state explicit at every recoverable boundary.
  • Correlate every observation to the proposal that caused it.
  • Gate the exact arguments immediately before execution.
  • Verify outcomes from environment state when possible.
  • Distinguish answer completion, task success, denial, escalation, and exhaustion.
  • Record enough typed evidence to reproduce a decision without logging secrets or hidden reasoning.

They also expose the remaining work. Chapter 17 designs tool surfaces, workspaces, approval binding, and the harness. Chapter 18 owns cross-session memory policy. Chapter 19 shows protocol and framework state machinery. Chapter 24 develops the full action-security architecture. Chapter 26 makes effects durable across crashes. The raw loop is valuable because those later abstractions can be judged against a mechanism the reader already understands.

  4. Restore the gate before continuing.

Next, add one outcome verifier to compare_modes() rather than trusting Stop.ANSWERED. The shipped fixture already reports task_success from the exact receipt. Add a second condition that the lookup observation preceded the effect, then make the router violate it. You should see control-flow completion remain answered while task success becomes false.

For an optional live adapter, implement Strategy.decide() with raw OpenAI-compatible HTTP against a local vLLM endpoint. Serialize only RunState.task and typed observations; publish the tool schemas; parse either one tool call or a final answer into the existing union. Do not change run_agent, ToolSpec, the gate, or the tests. The adapter is successful when replacing the deterministic strategy cannot bypass a single invariant.

16.12.1 Acceptance checks

  • All four shipped tests pass.
  • A denied proposal leaves effects empty.
  • A wanderer stops after exactly the configured number of decisions.
  • Every observation has a call identifier present in a prior proposal.
  • The live adapter, if added, changes only proposal selection.
  • Task success is computed from fixture state, not the final prose.

16.12.2 Honesty note

This build is an in-process control kernel. It does not provide authenticated tenant context, distributed cancellation, durable checkpoints, exactly-once effects, sandbox isolation, schema evolution, concurrent branch merging, human approval, or remote-effect reconciliation. The code is complete for the invariants it claims and deliberately silent about guarantees it does not implement.

16.13 What endures, what changes

What endures. A model proposes; code grants authority. State, observations, budgets, stop reasons, and success predicates should be explicit. Denials and expected failures are information. A workflow and an agent can share the same execution kernel; the difference is who selects transitions. Task success should be grounded in the environment rather than inferred from confident language.

What changes. Provider tool-call envelopes, model reasoning surfaces, context limits, agent SDK APIs, and orchestration products will keep changing. Frameworks will package different combinations of harness, runtime, memory, tracing, and deployment. Those changes do not remove the need to locate the proposal boundary, the policy decision, the effect path, and the outcome verifier.

16.14 Exercises

  1. Trace by hand. For Limits(max_steps=3, max_cost_units=4), write the exact stop reason and counters for a strategy that proposes two successful reads and then a final answer. Run it and reconcile any difference.
  2. Schema before policy. Propose {"order": "A-17", "amount_cents": 4999}. Verify that exact-field validation denies it before the policy function. Add a range constraint without mixing it into business authorization.
  3. Detect oscillation. Defeat the current no-progress detector by alternating between refunds of 9,998 and 9,999 cents. Extend detection to a bounded recent window while allowing one retry after a transient observation.
  4. Add parallel reads. Add a code-owned fan-out strategy for order and policy reads. Preserve call correlation when observations arrive in reverse order. Do not create a second execution kernel.
  5. Separate completion from success. Make a strategy answer “refund complete” without calling the refund tool. Add an outcome verifier that rejects it and a process verifier that explains the missing step.
  6. Design defense. A reviewer proposes moving the 5,000-cent rule into the system prompt so policy can change without deployment. Defend or reject the proposal, identify a safe configuration boundary, and name the residual stale-policy risk.

16.15 Notes and sources

The workflow-versus-agent distinction and “start with the simplest sufficient pattern” are consistent with Anthropic’s Building effective agents. The interleaving of reasoning, action, and observation originates in the ReAct paper. The practical emphasis on owning control flow and keeping state explicit is also developed in 12-Factor Agents.

Those sources motivate patterns; the executable specification, effect-boundary invariant, typed endings, and one-kernel comparison in this chapter are the book’s organizing treatment. Framework-specific execution begins in Chapter 19 after the raw mechanism is established.
