10.3 Goodput: the objective a serving system optimizes

A model endpoint reports record token throughput after a deployment change. At the same time, interactive users see more long pauses before the first token, and streaming occasionally freezes behind large prompts. Both observations can be true. The server completed more work per second by accepting larger batches, but too many individual requests missed the latency contract. Throughput rewarded work that users considered late.

This chapter builds the ability to explain and control that failure. The objective is goodput: the rate of requests that complete while satisfying every declared service-level objective (SLO). Let a measurement window have duration W seconds and contain requests indexed by i. Let T_i^{first} be request i’s time to first token (TTFT), T_i^{token} its time per output token (TPOT), and let S^{first} and S^{token} be their respective limits. A two-SLO goodput is

G(S^{first},S^{token})= \frac{1}{W}\sum_i \mathbf{1}\!\left[T_i^{first}\le S^{first} \;\land\; T_i^{token}\le S^{token}\right] \quad\text{requests/second}.

The indicator contributes one only when both conditions hold. A production contract might add availability, output validity, or task quality. Reporting each pass rate separately can hide that different requests fail each objective. The window and workload distribution are part of the definition.

TTFT and TPOT retain the boundaries from Chapter 9. End-to-end TTFT includes admission, queueing, tokenization, prefill, the first decode step, detokenization, and network delivery unless a narrower boundary is stated. TPOT describes gaps after the first token; tail inter-token latency is often more revealing than an average. The serving layer adds queueing and interference, so a fast isolated forward pass does not imply a fast service.

Goodput is a function of offered load. Batching initially improves useful work per invocation. Past a knee, queues grow, KV memory fills, preemption rises, and the qualified fraction collapses. Capacity is therefore a goodput-versus-arrival-rate curve for a pinned workload, accompanied by token lengths, concurrency, cancellations, cache pattern, and decoding policy.

Figure Figure 10.1 locates each decision. Components may share a process, but authentication, routing, scheduling, execution, KV ownership, streaming, and measurement remain distinct responsibilities.

Locate every optimization on this path and name its invariant. Batching can damage TTFT, cache affinity can overload a replica, and speculation can consume contested compute. Chapter 26 owns monetary cost and fleet provisioning; this chapter supplies their goodput mechanics.

10.5 Continuous batching and paged KV

A fixed batch of four decode requests has remaining lengths 2, 12, 4, and 10 tokens. After the two-token request finishes, its slot sits unused until the twelve-token request finishes. If every batch member reserves its maximum sequence-sized KV region, short requests also leave holes in device memory. Compute capacity and memory capacity are stranded for different versions of the same reason: variable-length requests are treated as fixed rectangles.

For a static batch of size B with output lengths L_1,\ldots,L_B, a simple slot utilization is

U_{static}=\frac{\sum_{i=1}^{B}L_i}{B\max_i L_i}.

The numerator is useful request-steps. The denominator is the slots available while the longest member runs. This equation ignores kernel efficiency and prefill, but it exposes length skew. Grouping similar lengths improves the ratio only when arrival rate and waiting-time budgets permit delaying requests to form those groups.

Continuous batching, also called iteration-level or in-flight batching, makes admission a per-iteration decision. After each decode step, the scheduler removes completed or cancelled requests and admits waiting work into the freed capacity. Orca established iteration-level scheduling with selective batching; its durable contribution is the scheduling boundary, not one queue policy. A production scheduler usually enforces a token budget because one long prefill and many one-token decode steps have very different work despite being the same number of requests.

The scheduler also needs a KV admission bound, fairness policy, cancellation path, and response when unknown output outgrows its reservation. Preemption may swap KV, discard it for recomputation, or reject work. Count preemptions, recomputed tokens, swapped bytes, queue age, and deadline misses; repeatedly evicting long requests can produce high utilization and poor goodput.

Long prefills create another head-of-line problem. Chunked prefill divides a prompt into bounded chunks interleaved with decode. Smaller chunks protect TPOT but add overhead and may delay that prompt’s TTFT. Chunk size is an SLO dial, not a copied constant.

Memory becomes schedulable when the KV cache is paged. A request sees logical KV blocks in token order; a block table maps them to noncontiguous physical frames. Frames are allocated only as tokens arrive. External fragmentation from requiring one contiguous maximum-length region disappears, while internal waste is bounded by less than one block per live sequence for a conventional tail block. The attention kernel follows the block table when reading keys and values. This is device-memory management inspired by virtual memory; it does not imply that an operating system is faulting model state to disk.

Paged blocks support copy-on-write: branches share immutable full prefix blocks but receive private frames when appending to a partial tail. Figure Figure 10.2 separates logical order from physical placement.

10.6 Prefix caching and cache economics

An agent sends the same system instructions, tool schemas, and policy text on every turn. The service recomputes their keys and values each time because one request changed a timestamp near the front of the prompt. A different deployment gets frequent cache hits, but accidentally lets two tenants share a prefix entry. Prefix caching is valuable only when identity, placement, and economics are all correct.

A prefix cache retains KV blocks after their original request no longer needs them, allowing a later request with the exact same token prefix and model state to skip the corresponding prefill. “Exact” is mechanical. Semantically equivalent text with different token IDs is a miss. Equal token IDs computed under different model weights, tokenizer behavior, position treatment, prompt template, multimodal input, or adapter are not interchangeable.

The cache key must therefore cover every input that can change KV values. The lab uses

K=(tenant,model,tokenizer,template,adapter,H(token\ IDs)).

The tenant component is policy, not mathematics. Explicit cross-tenant sharing of public immutable prefixes needs its own trust domain and auditable opt-in. Default isolation prevents a hit from becoming a presence oracle or state leak.

Caching has a break-even point. Let C be the cost to recompute a prefix, C_w the cost of the first computation plus writing the cache entry, C_r the cost of one later read, and r the number of future reuses. Without caching, total cost is (1+r)C. With caching, it is C_w+rC_r. When C_r<C, caching wins if

r>\frac{C_w-C}{C-C_r}.

“Cost” may be milliseconds on the critical path, accelerator-seconds, energy, or an internal price, but every term must use the same unit. If a remote read is slower than recomputation, no reuse count repairs the critical-path latency unless the read can be overlapped or frees a more constrained resource. The fixture assigns C=1, C_w=1.25, and C_r=0.10 cost units, producing a threshold of 0.278 future reuses. Because reuse arrives in whole events, the first reuse pays back the write in that fixture.

For eviction, approximate value density as expected reuse probability times recomputation saved, divided by retained bytes. Least-recently-used is cheap when recency predicts reuse but fails on periodic jobs, scans, and large popular prefixes. Size- and recomputation-aware policies protect denser entries; pinned bytes need quotas.

A radix tree exposes shared paths; parent-linked block hashes expose equivalent reuse. Neither places work across replicas. A KV-aware router balances reusable length against queue depth, free frames, and deadline. Always choosing the largest hit can create a hot replica, so route on predicted completion or SLO risk.

Hierarchical KV caches extend the working set to host memory, peers, storage, or a shared service, adding transfer and coordination. Track usable-hit bytes, read latency, transferred bytes, recomputed tokens, and TTFT; lookup hit rate alone can mislead. Compressed or low-precision KV also enters Section Section 10.8’s validation ladder.

Prompt ordering affects reuse, but Chapter 13 owns that caller-side layout and compaction. The server hashes what it receives and never assumes semantic equivalence.

10.7 Speculation and constraints trade compute for serial steps

Autoregressive decode waits for one target-model decision before constructing the next input. Two serving techniques reduce the cost of that serial boundary in different ways. Speculative decoding guesses several future tokens and verifies them together. Constrained decoding removes illegal next tokens before sampling. Both can improve a target metric, and both can waste work when their assumptions are wrong.

In exact speculative sampling, a fast draft distribution q proposes a token x, and the target distribution p accepts it with probability

a(x)=\min\!\left(1,\frac{p(x)}{q(x)}\right).

On rejection, sample from normalized residual (p-q)_+, where (v)_+=\max(v,0). This correction emits distribution p. The expected one-token acceptance is

\sum_x q(x)a(x)=\sum_x\min(p(x),q(x))=1-\operatorname{TV}(p,q),

where \operatorname{TV} is total-variation distance. For a block, the target scores several proposals in parallel, accepts the valid prefix, then corrects if needed. Exactness covers the target distribution under compatible processors, not identical random traces or latency.

Speedup is accepted tokens per verification divided by all added work. Draft compute, KV, shape-dependent verification, and rejected proposals consume resources. At saturation, an isolated latency win can reduce fleet goodput. Measure proposal acceptance, draft/verification time, memory, and goodput. Draft models, prediction heads, prompt lookup, and n-grams share this accounting boundary.

Constrained decoding starts from a language recognizer rather than a predictor. A finite-state machine (FSM) represents regular constraints such as fixed choices and many lexical patterns. A context-free grammar (CFG) adds a stack, allowing nested structures such as balanced arrays and objects. At each step, the runtime advances a parser state, determines which token byte strings can continue a valid parse, masks every other token, renormalizes, and samples. Tokenization makes this harder than checking characters after generation: one token can contain several characters, split a delimiter, or cross a grammar transition.

Masks can be compiled and cached by parser state; deterministic literals may jump forward without a model call. An empty allowed set indicates an unsatisfiable schema or implementation error. Dropping the constraint changes the contract.

A grammar guarantees syntax, not truth, authorization, or business validity. Chapters 17 and 24 validate values and authorize effects. If illegal tokens hold mass m, masking removes m and renormalizes; the lab removes 0.60. This is exact for the constrained distribution, not unconstrained generation. Because format restrictions can impair reasoning, evaluate designs that separate deliberation from a small validated final object.

10.8 Quantization is a validation problem

A four-bit checkpoint occupies less storage than its sixteen-bit parent, yet serves fewer requests per second. Its engine dequantizes weights through an unfused fallback kernel, while an important tool-selection task regresses even though perplexity barely moves. “Four bit” described storage, not the complete numerical and execution contract.

Quantization approximates high-precision values with a smaller representation. Describe a recipe along several independent axes:

  • what is quantized: weights, activations, KV cache, or communication buffers;
  • numerical family and width: integer, floating point, or another codebook;
  • scale granularity: tensor, channel, group, row, or block;
  • when scales are selected: offline/static or per-input/dynamic;
  • accumulator and output precision; and
  • kernel, hardware, and model-architecture support.

W4A16 means four-bit weights and sixteen-bit activations; W4A4 compresses both. Weight-only recipes target decode weight bandwidth. Activation quantization enables lower-precision matrix multiplication but faces input-dependent outliers. KV error may accumulate with context length.

For symmetric integer quantization of a block x, choose scale s>0 and signed integer bound Q. The lab uses

q_i=\operatorname{clip}\!\left(\operatorname{round}(x_i/s),-Q,Q\right), \qquad \hat{x}_i=sq_i.

With b bits, a common symmetric bound is Q=2^{b-1}-1. Smaller blocks give each local range its own scale and reduce damage from distant outliers. They also store more scale metadata. If each block contains g values, value width is b bits, and each scale uses b_s bits, a basic storage estimate is

b_{effective}=b+\frac{b_s}{g}

bits per value, before alignment, zero points, global scales, and container overhead. “Four bit” can therefore mean 4.06, 4.25, 5.0, or another effective width depending on block size and metadata.

Outliers motivate several recipes. Weight-only methods use approximate second-order information or protect activation-salient channels. Activation-aware methods can migrate scale difficulty: for invertible diagonal D,

XW=(XD^{-1})(DW).

Rescaling eases activation range while moving it into offline-calibrated weights. Other recipes isolate, decompose, or rotate outliers. The full-precision algebra may be exact; quantization creates the approximation. Calibration must match deployed data and shapes.

Validate from representation to service: loading and scale layout; optimized kernels for real shapes and feature combinations; layer error and perplexity diagnostics; task, tool, structured-output, long-context, safety, and calibration metrics; then cold/warm latency, memory, energy, and goodput. Predeclare detectable quality regressions.

The lab places two outliers in 256 synthetic values and applies symmetric four-bit quantization. Blocks of 16, 64, and 256 yield RMSE 0.455, 0.978, and 1.415 fixture units, while their simple effective widths are 5.00, 4.25, and 4.06 bits per value. The smaller block buys fidelity with metadata. These values teach the trade-off; they do not rank production formats.

10.9 Disaggregate when interference exceeds transfer

A decode stream produces a token every few milliseconds until a long prompt enters the same worker. The prompt’s prefill turns one scheduler iteration into a large compute job, and every active stream stalls. Chunking reduces the stall but cannot give prefill and decode different hardware shapes or independent scaling. Separating the phases removes interference, then adds a KV transfer to the critical path.

In prefill/decode disaggregation, a prefill worker processes the prompt and materializes its KV state. That state moves to a decode worker, which generates output and streams it to the caller. Prefill is commonly more compute-oriented; decode is commonly more memory-bandwidth- and KV-capacity-oriented. Separate pools can use different parallel widths, batch policies, and replica counts, and can scale according to input-token and output-token demand.

The split is not automatically faster. Let \Delta I be latency or capacity recovered by removing prefill/decode interference and \Delta S the benefit of phase-specialized hardware or batching. Let T_{KV} be nonoverlapped KV-transfer time, T_R routing and coordination, T_Q extra queue imbalance, and T_F expected failure/retry cost. A useful admission inequality is

\Delta I+\Delta S>T_{KV}+T_R+T_Q+T_F.

If M_{KV} bytes must cross a path with effective bandwidth B_{eff} bytes per second, then T_{KV}\ge M_{KV}/B_{eff} before protocol overhead. Long prompts create more interference to remove but also more KV state to transfer. Layer-wise streaming can overlap transfer with prefill; shared memory or a remote cache may avoid a full copy; congestion can make effective bandwidth much lower than a link’s headline. Measure the critical path rather than substituting a fabric specification.

Figure Figure 10.3 makes ownership transitions explicit. The router needs cancellation semantics spanning both pools. A decode failure after successful prefill should not automatically recompute the prompt if a valid, authorized KV object remains available. Conversely, a stale or partial KV object must never be treated as complete.

sequenceDiagram
    participant C as Caller
    participant R as SLO-aware router
    participant P as Prefill pool

Chunked prefill and disaggregation are complementary controls. An aggregated worker can chunk to cap decode stalls. A prefill pool can chunk for fairness among long prompts and to overlap KV transfer. Tiny chunks increase launch and scheduling overhead; huge chunks recreate head-of-line blocking. Tune against TTFT and TPOT together.

Capacity planning often describes xP yD: x prefill replicas feeding y decode replicas. The ratio depends on input/output lengths, prefix-hit rates, reasoning budgets, parallel topology, and SLOs. Static ratios fail when traffic composition changes, so a controller may reassign roles or route selected short prompts through aggregated workers. The lab’s simple inequality rejects disaggregation for a 0.25 GiB KV fixture that saves 3 ms but pays 4.05 ms, and accepts it for a 4 GiB fixture that saves 35 ms while paying 22.8 ms. Both results depend on declared synthetic bandwidth and coordination costs.

10.10 Parallelism and multi-tenancy share more than weights

When one model does not fit or one replica cannot meet demand, “use more GPUs” leaves the important choice unspecified. Parallel strategies move different data at different frequencies. Serving magnifies their communication costs because decode repeatedly runs small, latency-sensitive steps rather than large training batches.

Strategy What is divided Serving advantage Dominant serving risk
Data parallel replicas requests, with complete model copies independent scaling and failure domains load and prefix-cache imbalance
Tensor parallelism (TP) tensors within each layer makes each layer fit; combines device bandwidth collective communication on every layer and decode step
Pipeline parallelism (PP) consecutive layer groups crosses nodes with fewer, larger transfers than TP pipeline bubbles, stage imbalance, and per-request latency
Expert parallelism (EP) mixture-of-experts expert weights distributes a sparse model’s expert memory and work token all-to-all, hot experts, and synchronization at imbalance

TP shards matrix work and reduces partial results with collectives, favoring fast scale-up links; communication eventually dominates small decode steps. PP sends activations between layer stages but irregular microbatches leave bubbles. EP routes tokens to experts; sparse activation lowers arithmetic, not expert storage or all-to-all cost, and hot experts set batch time.

Serving carries KV state and deadlines instead of optimizer state. Parallel width changes KV placement, cache reuse, verification shapes, and failure radius. Benchmark the actual topology and collectives; an oversubscribed boundary can erase arithmetic gains.

Multi-LoRA shares expensive xW while applying adapter-specific low-rank xAB terms on the fly, avoiding a merged W per tenant. Adapters can page from host, and grouped kernels batch heterogeneous terms, as Punica and S-LoRA show. Account for rank, load latency, residency, and TP compatibility.

Sharing a base model, process, accelerator, or cache expands the security boundary. A minimum serving contract includes:

  • cache keys containing tenant policy and every model/tokenizer/template/adapter revision;
  • authenticated adapter identity, integrity verification, and an authorization check before load;
  • quotas for KV bytes, prefix entries, active sequences, speculative work, and reasoning tokens;
  • explicit release and zeroization behavior for host buffers and accelerator allocations;
  • isolation tests for local/scratch memory, not only global device memory;
  • constant-work or coarse-bucket mitigations where cache hits, adapter loads, or queue timing reveal sensitive presence; and
  • attestation when the threat model requires proof of the loaded software/firmware measurement.

Attestation establishes a measured starting stack, not correct authorization or absence of memory and timing leaks. Zeroization must cover cancellation, crashes, and eviction. Stronger tenants may require separate devices, nodes, encryption domains, or deployments.

LeftoverLocals recovered data across GPU process boundaries on affected implementations. A container boundary is not evidence that device-local memory is scrubbed. Chapter 24 owns the full threat model; serving owns these shared-state facts.

10.11 Choose an engine by its feature lattice

An engine wins a public throughput benchmark, so a team selects it for a service that needs one quantized mixture-of-experts model, two LoRA adapters, grammar constraints, prefix caching, tensor parallelism, and speculative decoding at the same time. Each feature appears in documentation. Their combination falls back to an unfused path or is unsupported. Feature checklists are one-dimensional; deployments occupy intersections.

A feature lattice is the set of combinations that work for a pinned model, hardware topology, engine revision, and request shape. Engine selection begins with a workload card:

Workload dimension Values to pin before comparison
Model architecture, exact revision, tokenizer/template, dense or MoE, modalities

The PagedAttention work associated with vLLM supplies a useful reference architecture: iteration scheduling plus block-managed KV allows high-concurrency autoregressive serving without contiguous per-sequence reservations. That mechanism is durable. An engine comparison still needs to verify model-specific attention kernels, graph capture, compilation, quantization layouts, distributed collectives, grammar integration, speculative strategy, and adapter kernel support.

Optimized engines specialize by shape. Kernel fusion avoids round trips to device memory. CUDA graphs or analogous capture mechanisms reduce launch overhead for supported static patterns. Compilers specialize operations and layouts. These optimizations create warm-up time and shape coverage boundaries. A rare batch size, long context, adapter rank, or processor can trigger recompilation or fallback. Record warm-up policy and count fallback paths. An OpenAI-compatible HTTP route provides interface portability, not identical tokenization, sampler order, stop behavior, usage accounting, or streaming events.

Not every language model has the autoregressive execution assumed so far. A diffusion language model may iteratively revise masked positions with bidirectional attention and commit several positions in parallel. A conventional causal KV cache cannot be reused unchanged when previously hidden states can change. Scheduling units become denoising steps or blocks rather than one appended token. Approximate caches and confidence-guided commits introduce quality/throughput controls. Treat such models as a different engine capability, not as a new checkpoint behind an autoregressive server flag.

Cluster orchestration sits above the model engine. A multi-node replica needs gang placement and topology awareness so all shards start on compatible links. Readiness begins after weights, kernels, and communication groups are usable, not when a container starts. Scale-down drains streams and releases or migrates KV state. Rolling a model, tokenizer, template, quantization recipe, or adapter changes cache identity. Autoscaling signals should include queued input/output tokens, active KV bytes, cache hits, preemptions, and SLO risk; CPU utilization misses the constrained resource.

The evaluation procedure is direct: create a matrix whose rows are workload slices and whose columns are required feature combinations. Run correctness gates first, then sweep offered load and measure goodput, memory, warm-up, and failure recovery. Headline benchmark results can shortlist engines. Only the pinned lattice can select one.

10.12 Production test-time compute needs a budget

A reasoning model improves a difficult planning task when allowed to think longer, so a service enables the highest reasoning setting for every request. Simple requests now occupy decode slots and KV frames for many more tokens. Queueing pushes hard requests past their deadlines, and total successful work falls. More compute improved isolated accuracy while reducing production utility.

Chapter 8 treated test-time compute as search and verification. In serving, reasoning budget is a reservation: reasoning tokens, effort tier, deadline, candidates, or tool calls. These are not interchangeable. Completion limits can truncate answers, effort can be opaque, and verifier stopping does not bound each attempted trajectory.

Hidden reasoning still consumes decode steps and holds state when only a final answer or summary returns. Use reported reasoning usage or empirical latency/token distributions by tier. Do not log private chain of thought for observability: summaries are generated output, not execution proof. Chapter 25 owns faithfulness; telemetry should prefer budgets, timings, tool events, verifier outcomes, and evidence.

A budget policy compares marginal value with marginal system cost. For one more reasoning allocation, let \Delta p be the estimated increase in task success, V the value of success, \Delta n additional generated tokens, c_n their direct unit cost, \Delta h added KV/slot holding time, c_h its capacity shadow price, and D expected harm from delay. Continue only while

\Delta p\,V > \Delta n\,c_n+\Delta h\,c_h+D.

The shadow price c_h rises near saturation, making budgets load- and difficulty-aware. Start cheap, escalate when a classifier or verifier predicts value, and enforce hard deadlines and token ceilings. Record tier, stop reason, outcome, and queue conditions; lower budgets under overload before rejecting all traffic.

Tools expose holding cost. Keeping KV through a tool wait avoids recomputation but pins memory; eviction requires re-prefill; a slower tier pays transfer. Decide from expected wait, state size, reuse, and queue pressure. Cancellation spans tool, model, and KV lease. Stable tool definitions also create reusable prefixes.

Inference-time budget forcing stops or extends a trace; s1 used it as an experimental control. Training-time Length Controlled Policy Optimization teaches requested lengths. Truncation can cut solutions, continuation can overthink, and trained controllers need separate evaluation. None guarantees monotonic quality.

The lab’s synthetic curve rises from score 0.631 at 32 reasoning tokens to 0.774 at 256, then declines to 0.771 at 512 while score per thousand tokens falls throughout. It represents saturation and an overthinking tail, not a model measurement. A production curve must be estimated by task slice, outcome severity, and load. The serving decision is to purchase marginal success, not to maximize visible reasoning length.

10.13 Landscape 2026 (dated)

Verified on 2026-07-19. Current vLLM prefix-caching documentation describes block hashing with LoRA, multimodal, and cache-salt identity. Diffusion-language-model serving remains an emerging, separate execution path. Fast-dLLM studies approximate KV reuse and confidence-aware parallel decoding for bidirectional diffusion models. 10.15 What endures, what changes

What endures. Goodput counts work that satisfies the complete user-visible contract. Continuous scheduling converts length variation into iteration-level admission; paged KV converts sequence growth into block allocation. Prefix reuse requires exact state identity and must beat recomputation after read, write, placement, and eviction costs. Exact speculative sampling preserves a target distribution through accept/reject correction but can lose under contention. Grammar constraints guarantee syntax, not semantics or authority. Quantization is specified by object, format, scale, accumulator, kernel, and validation—not bit width alone. Disaggregation wins only after subtracting KV movement and coordination. Parallelism exchanges memory and arithmetic for communication. Shared weights, adapters, caches, and accelerators are security boundaries. Reasoning consumes tokens, KV, slots, and deadlines even when its trace is hidden.

What changes. Engine versions, supported model-feature combinations, kernel coverage, accelerator formats, graph and compiler behavior, quantization recipes, speculative strategies, diffusion-model runtimes, remote-KV protocols, Kubernetes APIs, orchestration projects, provider effort controls, prices, and benchmark results all move. Keep these in pinned adapters, compatibility tests, and the dated landscape. Removing the landscape leaves every decision inequality and invariant above intact.

Chapter 11 moves from serving many fixed model variants to creating and governing customized ones; the serving lattice here becomes the deployment constraint for those artifacts.
